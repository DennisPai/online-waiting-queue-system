# 客戶永久資料庫 PRD

## 背景與目標

修玄宮候位系統目前的資料生命週期僅限「單期」— 每期辦事結束後候位記錄被清空，客戶的歷史來訪紀錄完全消失。這導致：
- 無法辨識回頭客（每次來都當新客處理）
- 無法統計客戶來訪頻率和歷史
- 家庭成員關係無法跨期保留

**目標**：建立客戶永久資料庫，讓每一期的客戶資料自動累積，形成完整的客戶歷史紀錄。

**KPI**：
- 「結束本期」操作成功率 ≥ 99%（資料零遺漏）
- 新舊客識別準確率 ≥ 95%（基於姓名 + 農曆生日比對）
- 管理員可在 3 秒內搜尋到任一客戶的歷史紀錄

---

## 角色與權限

| 角色 | 權限 |
|------|------|
| 管理員 | 「結束本期」、客戶資料庫搜尋/檢視/編輯、家庭分組管理、標籤管理、備註 |
| 訪客 | 無權限（客戶資料庫僅限管理員存取） |

## 核心概念

### 客戶識別
- **唯一識別方式**：姓名 + 農曆生日（name + lunarBirthYear/Month/Day）
- 同名同農曆生日 → 視為同一人
- 「結束本期」時自動比對，若已存在則更新；若不存在則建立新客戶

### 家庭（Household）分組
- **自動分組**：地址完全相同的客戶自動歸為同一家庭
- **手動調整**：管理員可手動合併、拆分、或重新分配家庭成員
- 家人 = 獨立客戶，各自擁有完整的客戶資料和來訪紀錄

### 獨立資料庫
- 資料庫名稱：`customer_archive`（與候位系統 `queue_system` 分開）
- **永久資料庫不設一鍵清除功能**（防止誤操作）

## 資料結構

### Customer（客戶）
| 欄位 | 類型 | 說明 |
|------|------|------|
| _id | ObjectId | 自動生成 |
| name | String | 姓名（必填） |
| lunarBirthYear | Number | 農曆出生年（必填） |
| lunarBirthMonth | Number | 農曆出生月（必填） |
| lunarBirthDay | Number | 農曆出生日（必填） |
| solarBirthDate | Date | 國曆出生日期（自動轉換） |
| zodiac | String | 生肖（自動計算） |
| phone | String | 電話 |
| address | String | 地址 |
| householdId | ObjectId | 所屬家庭 ID |
| tags | [String] | 標籤（管理員自訂） |
| notes | String | 管理員內部備註 |
| totalVisits | Number | 累計來訪次數 |
| firstVisitDate | Date | 首次來訪日期 |
| lastVisitDate | Date | 最近來訪日期 |
| createdAt | Date | 建立時間 |
| updatedAt | Date | 更新時間 |

### Household（家庭）
| 欄位 | 類型 | 說明 |
|------|------|------|
| _id | ObjectId | 自動生成 |
| address | String | 住家地址 |
| memberIds | [ObjectId] | 成員的 Customer ID |
| createdAt | Date | 建立時間 |
| updatedAt | Date | 更新時間 |

| 欄位 | 類型 | 說明 |
|------|------|------|
| _id | ObjectId | 自動生成 |

| remarks | String | 備註（從候位記錄帶入） |
| familyMembers | [String] | 本次同行家人姓名 |
| createdAt | Date | 建立時間 |

## 功能需求

### Phase 1：結束本期 + 資料歸檔

#### F1-1「結束本期」按鈕
- **位置**：管理後台，取代現有的「清空全部」功能
- **觸發方式**：管理員手動點擊
- **操作對象**：所有非已取消的候位記錄

#### F1-2「結束本期」執行流程
1. 點擊按鈕 → 彈出確認視窗（「確定要結束本期？本期 N 位客戶的資料將歸檔至永久資料庫。」）
2. 確認後開始處理，顯示進度
3. 遍歷本期所有非已取消的候位記錄：
   - **比對客戶**：用「姓名 + 農曆生日」在 `customer_archive` 搜尋
   - **新客**：建立 Customer + VisitRecord + 處理家庭分組
   - **舊客**：更新 lastVisitDate、totalVisits +1、寫入新 VisitRecord
   - **家人處理**：候位記錄中的家人也各自作為獨立客戶處理（同樣的新舊客邏輯）
   - **備註/問事主題**：從候位記錄帶入 VisitRecord
4. 家庭自動分組：地址完全相同的客戶自動歸入同一個 Household
5. 清空候位資料
6. 顯示統計報告：
   - 本期總人數
   - 新客 N 人
   - 回頭客 N 人
   - 新建家庭 N 組

#### F1-3 新舊客比對規則
- **完全匹配**：姓名 + 農曆年 + 農曆月 + 農曆日都相同 → 視為同一人
- **姓名相同但生日不同** → 視為不同人（建立新客戶）
- **生日相同但姓名不同** → 視為不同人（建立新客戶）
- 比對時忽略姓名前後空格

### Phase 2：客戶資料庫管理介面

#### F2-1 客戶列表頁
- 搜尋：支援姓名、電話、地址搜尋
- 篩選：按標籤篩選
- 排序：按姓名、來訪次數、最近來訪日期
- 列表欄位：姓名、生肖、電話、來訪次數、最近來訪日期、標籤

#### F2-2 客戶詳情頁
- 基本資料（可編輯）
- 來訪歷史（VisitRecord 列表，按日期倒序）
- 所屬家庭（顯示同一 Household 的其他成員）
- 標籤管理（新增/移除）
- 備註（可編輯）

#### F2-3 家庭管理
- 自動分組顯示（基於地址）
- 手動合併：選擇兩個或以上客戶 → 合併為同一家庭
- 手動拆分：將客戶從家庭移出
- 家庭列表檢視：按地址分組，顯示成員

## 資料流向

```
候位登記 → WaitingRecord (queue_system)
                    │
            管理員點擊「結束本期」
                    │
                    ▼
        遍歷所有非已取消記錄
                    │
            ┌───────┴───────┐
            ▼               ▼
         新客戶           舊客戶
     建 Customer      更新 Customer
     建 VisitRecord   建 VisitRecord
            │               │
            └───────┬───────┘
                    ▼
          家庭自動分組（依地址）
                    │
                    ▼
            清空候位記錄
                    │
                    ▼
            顯示統計報告
```

---

## 非功能需求

- **資料安全**：客戶永久資料庫僅限管理員存取，不對外暴露 API
- **效能**：客戶搜尋有索引（name、phone、address、householdId），搜尋回應 < 500ms
- **資料完整性**：「結束本期」為原子操作 — 全部成功或全部回滾，不允許部分歸檔
- **向下相容**：不影響現有候位系統的任何功能

## 未來擴充規劃

### Phase 3：客戶統計儀表板
- 本月/本季/本年新客數、回頭客數、回頭率
- 平均來訪間隔
- 熱門問事主題排行
- 客戶成長趨勢圖表
- 讓管理員一眼掌握「最近生意怎麼樣、客人都來問什麼」

### Phase 4：客戶回訪提醒
- 設定回訪週期（例如：超過 6 個月未來訪）
- 系統自動標記「待關懷客戶」
- 管理後台顯示提醒清單
- 適合需要定期追蹤的客戶

### Phase 5：客戶匯出/匯入
- 匯出客戶資料為 Excel（含來訪歷史）
- 從 Excel 匯入舊客戶資料（批量建檔）
- 適用於將紙本或其他系統的歷史客戶資料一次性匯入

### Phase 6：諮詢歷程詳細化
- VisitRecord 擴充欄位：處理方式、建議、後續追蹤事項
- 形成完整的諮詢歷程紀錄（類似病歷）
- 下次客戶來訪時可快速回顧上次的建議和追蹤事項
- 支援管理員在辦事後補填或隨時更新

## 風險

| 風險 | 影響 | 應對方案 |
|------|------|----------|
| 新舊客比對失誤（同名不同人、改名等） | 資料合併錯誤 | 管理員可手動修正；Phase 2 提供合併/拆分功能 |
| 「結束本期」中斷（網路斷線、伺服器異常） | 部分資料遺漏 | 原子操作確保全部成功或回滾；可重新執行 |
| 家庭自動分組誤判（地址微小差異） | 分組不正確 | 管理員手動調整；未來可加模糊匹配 |
