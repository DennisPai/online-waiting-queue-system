# 修玄宮候位系統 — 外部客戶使用情境

> 對象：一般民眾、宮廟信眾（未登入 / 訪客身分）。
>
> 來源：根據前台前端頁面（`HomePage` / `RegisterPage` / `StatusPage` / `LoginPage`）、`docs/PRD.md`、`docs/USER_GUIDE_ADMIN.md`、以及 2026-05-22 / 05-23 完成的 4 個 OpenSpec Change（Change A 候位一致性 / Change B 家人地址 / Change C 全系統農曆 + 移除前台取消 / 5/23 follow-up patches）整理。
>
> **角色限制（PRD 規範）**：訪客僅能登記候位、查詢候位，**不能修改資料**；前台家人上限 3 人；前台不再有「取消預約」按鈕（Change C 已移除，誤點誤取消的歷史問題終結）；生日全系統限填農曆（Change C + follow-up #5）。
>
> **進入點總覽**：
> - 直接訪問首頁 `https://openqueue-test.zeabur.app/`
> - 「我要候位」→ `/register`
> - 「查詢候位」→ `/status/search`
> - 候位成功後跳轉 `/status/:queueNumber`
> - 點頁尾「管理員登入」→ `/login`（雖然是後台入口，列在管理員文檔）

---

## 情境 1：首次訪問前台首頁（公告期 / 候位未開）

- **使用者目標**：第一次到網站，想知道修玄宮在做什麼、下次什麼時候辦事、要不要報名。
- **進入點**：直接打 URL 或從 LINE / FB 連結進入 `/`。
- **操作步驟**：
  1. 開啟 `https://openqueue-test.zeabur.app/`。
  2. 看頁面標題「修玄宮天上聖母開科辦事」、副標「透過網路輕鬆登記候位，不必排隊等待」。
  3. 往下捲看「使用步驟」三步教學（線上登記 / 預計時間 / 提早 10 分鐘到場）。
  4. 視管理員設定，可能看到「我要登記候位」+「查詢候位」兩張卡片，也可能只看到「查詢候位」（取決於 `publicRegistrationEnabled`）。
- **預期結果**：
  - 看得到下次辦事時間（透過 `QueueStatusDisplay` 顯示 `nextSessionDate`）。
  - 若 `publicRegistrationEnabled=true` → 顯示「我要候位 / 立即候位」按鈕。
  - 若 `publicRegistrationEnabled=false` → 隱藏報名按鈕，僅顯示查詢卡片，避免在開放時段外被報名。
  - 若管理員設定了活動 banner（`eventBanner.enabled=true`）→ 顯示自訂活動報名區塊。
- **邊角案例**：
  - **未設定 `nextSessionDate`**：時間欄會顯示「未設定」，使用者只能查詢，無法判斷何時辦事。
  - **`isQueueOpen=true` 但 `publicRegistrationEnabled=false`**：辦事中但不收新報名；首頁變成「目前叫號 N 號」大字 + 只能查詢。
  - **手機瀏覽**：字體大小用 responsive，最小 `xs: 2rem`，需測試 iPhone SE / 大螢幕對齊。
- **失敗處理**：後端 `/health` 不通或 Redux `queueStatus` 載入失敗 → 副標仍顯示，但時間欄空白、報名/查詢按鈕無法觸發；客戶無自助復原路徑，需聯絡宮廟。

## 情境 2：首頁查看「目前叫號」（辦事進行中）

- **使用者目標**：今天是辦事日，想知道現在叫到幾號、自己幾點能進去。
- **進入點**：辦事當天直接訪問 `/`，此時 `isQueueOpen=true`。
- **操作步驟**：
  1. 開啟首頁，最上方標題自動變成「目前叫號」+ 大號碼（如「12 號」）。
  2. 中間 Divider 下方顯示「下一位等待號碼：13」。
  3. 系統每 5 分鐘自動 `getQueueStatus` 更新一次（`isQueueOpen=true` 才啟用 polling）。
- **預期結果**：使用者一眼看到目前進度，可自行估算到場時間（每位 `minutesPerCustomer` 分鐘，預設 13）。
- **邊角案例**：
  - **5 分鐘輪詢沒抓到最新值**：使用者需手動重整 F5 或進「查詢候位」頁拿即時值。
  - **`currentQueueNumber=0`**：剛開始辦事但還沒叫號 → 顯示「0 號」可能造成誤會，應理解為「尚未開叫」。
- **失敗處理**：Socket.io 或 API 失敗 → 數字停留在最後一次成功值，無顯眼錯誤提示。

## 情境 3：報名候位（完整流程、簡化模式關閉）

- **使用者目標**：登記今天的辦事候位，把自己跟家人一起排入。
- **進入點**：首頁 → 點「立即候位」→ `/register`。
- **操作步驟**：
  1. 進入「候位登記」頁，看到頂部「候位提醒卡片」（如「目前已有 32 位候位，您的編號將會是第 33 號」）。
  2. **基本資料區塊**：填姓名（必填）、聯絡手機（必填）、電子郵件（選填）、性別（下拉：男/女/其他）。
  3. **農曆生日**：透過 `BirthdayPicker` 選農曆出生年（民國年）、月、日、是否閏月。**全系統強制農曆**（Change C v3+follow-up #5），介面標題改為「農曆生日」並提示「請先自行查好農曆生日」。
  4. **地址資訊區塊**：填主地址 + 選地址類別（住家 / 工作場所 / 醫院 / 其他）；可加多筆地址。
  5. **家人資訊區塊**：可加最多 3 位家人，每位含姓名、性別、農曆生日（含閏月）、地址 + 地址類別；可勾「同上（使用主客戶地址）」直接套用主客戶地址。
  6. **問事項目**：勾選諮詢主題（身體 / 運途 / 因果 / 家運祖先 / 事業 / 婚姻感情 / 學業 / 收驚加持 / 其他），勾「其他」會展開額外文字框（最多 500 字）。
  7. **備註欄位**：選填，最多 1000 字。
  8. 按「完成登記」按鈕，loading 旋轉到後端回應。
  9. 成功 → 自動跳轉 `RegistrationSuccess` 頁面，顯示「您的候位號碼：N」+ 前面等待組數 + 預估等待時間 + 預估輪到時間 + 第幾個排序。
- **預期結果**：
  - 後端 `validateRequiredFields` 確認：姓名、電話、農曆三欄（年/月/日）都有值（follow-up #3 補的必填驗證）。
  - 後端 `autoFillDates` 自動把農曆轉成國曆寫進 DB（民國年版 follow-up #5）。
  - 系統 atomically 用 `issuedCount` 閘門佔走一個名額（Change A），不會撞號、不會超收。
  - 跳轉到 `/status/:queueNumber` 等同記錄成功。
- **邊角案例**：
  - **超過 80 號**：候位提醒卡片會多出警告「超過 80 號預計將排至凌晨 1 點以後，若非重大問題急需求助，建議預約下次問事」。
  - **農曆閏月**：勾選閏月 checkbox 表示「閏 X 月」，後端 `lunarIsLeapMonth=true` 一併儲存。
  - **家人勾選「同上」但主地址尚未填**：FamilySection 顯示警示「請先填寫主客戶地址」並 disable checkbox。
  - **第 3 位家人加完**：「新增家人」按鈕消失（前端 `length < 3` 判斷）。
- **失敗處理**：
  - 必填漏填：對應 `formErrors` 顯示紅字提示，按鈕 disabled。
  - 額滿（issuedCount 已達上限）：後端回 friendly 訊息，前端跳 RegisterPage 邏輯 → 切到「本次報名已額滿」面板（見情境 8）。
  - 後端 5xx：alert 紅色錯誤，候位號碼未產生，使用者可重試。

## 情境 4：報名候位（簡化模式開啟）

- **使用者目標**：宮廟現場替長輩快速代填，只想留姓名、稍後補資料。
- **進入點**：管理員把 `simplifiedMode=true` → 前台 `/register` 表單驗證自動寬鬆。
- **操作步驟**：
  1. 進入 `/register`，所有欄位不再強制 required（`required={!simplified}`）。
  2. 只填姓名 → 點「完成登記」。
- **預期結果**：後端不擋缺電話、缺農曆等，仍能取得候位號碼。
- **邊角案例**：
  - **simplifiedMode 時仍有 lunar 必填？**：依 follow-up #3 規格，「簡化模式仍 skip」lunar 驗證 → 簡化模式可只填姓名。
  - 後台事後得補資料才能算 zodiac / virtualAge / 國曆生日。
- **失敗處理**：姓名空字串仍會被擋（姓名是 hard requirement）。

## 情境 5：報名成功後查看候位號碼

- **使用者目標**：剛送出表單，要記下號碼以便之後回查。
- **進入點**：報名完成 → `RegistrationSuccess` 元件。
- **操作步驟**：
  1. 看到大字候位號碼。
  2. 看到「前面還有 N 組 / 預估等待 N 分鐘 / 預估輪到時間 HH:mm / 您的排序第 N 位 / 候位提醒文字」。
  3. 可截圖保存或按「返回首頁」回 `/`。
- **預期結果**：使用者帶著號碼離開，需自行記住或截圖。
- **邊角案例**：
  - **使用者不小心關掉頁面**：可用情境 6「查詢候位」找回。
  - **預估時間隨叫號進度更新**：使用者再回查詢頁時看到的「前面還有」數字會降低。
- **失敗處理**：成功頁面沒提供 email/簡訊副本（系統現無 notification 機制）→ 客戶忘記號碼只能靠姓名+電話查詢。

## 情境 6：查詢候位狀態（姓名 + 電話）

- **使用者目標**：之前報名過，現在想知道叫到哪、還要等多久。
- **進入點**：首頁 →「查詢候位」按鈕 → `/status/search`。
- **操作步驟**：
  1. 進入「候位狀態查詢」頁。
  2. 在「姓名」欄填姓名（可填本人或家人），或在「電話」欄填登記時電話，或兩者同時填（精確匹配）。
  3. 按「查詢候位狀態」按鈕。
- **預期結果**：
  - 顯示找到的候位記錄（單筆或多筆）。
  - 每筆顯示：編號、狀態 chip（等待中 / 處理中 / 已完成 / 已取消）、目前叫號、前面還有 N 組、查看詳細資料按鈕。
  - 候位日期文字「您已報名成功，問事號碼會在接近 YYYY 年 M 月 D 日時公布」。
- **邊角案例**：
  - **同一家庭多筆**：同一電話可能有多筆候位，全部列出（陣列）。
  - **管理員關閉 `showQueueNumberInQuery`**：候位號碼欄位被隱藏，只顯示狀態 + 詳細按鈕。
  - **以家人姓名查詢**：後端 `searchQueueByNameAndPhone` 會掃描 `familyMembers.name`，可查到該家庭的記錄。
  - **姓名 + 電話兩個都不填就按查詢**：前端 warning 提示「請輸入姓名或電話其中一個進行查詢」。
- **失敗處理**：
  - 查無結果：紅色錯誤框「找不到符合條件的候位記錄」（依後端 error 訊息）。
  - API 失敗：紅框顯示 error，使用者可重試。

## 情境 7：查詢候位 → 看「詳細資料」對話框

- **使用者目標**：確認我之前填了什麼資料、家人是否都登記到。
- **進入點**：情境 6 找到候位 → 點該筆的「查看詳細資料」按鈕。
- **操作步驟**：
  1. 開啟對話框，標題「編號 N 詳細資料」。
  2. 看四大區塊：基本資料 / 出生資料（國曆 + 農曆民國年顯示，含閏月註記）/ 諮詢主題 / 備註 / 地址資訊 / 家人資訊。
  3. 看完按「關閉」回查詢頁。
- **預期結果**：純檢視，**無編輯按鈕**（PRD 規範：訪客僅能查看）。
- **邊角案例**：
  - **歷史紀錄缺農曆**：「出生日期：出生日期未設定」。
  - **沒有家人**：家人區塊整段不顯示。
  - **「其他」諮詢主題**：括號內顯示客戶填的 `otherDetails`，例如「其他(詢問搬家方位)」。
  - **地址類型對照**：home → 住家、work → 工作場所、hospital → 醫院、other → 其他。
  - **管理員關閉 `showQueueNumberInQuery`**：對話框標題的「編號 N」隱藏、只顯示「詳細資料」；開啟時才顯示完整「編號 N 詳細資料」（行為對齊情境 6 候位列表的編號隱藏邏輯）
  - **家人生日只顯示農曆**：家人段不顯示國曆生日（跟主客戶段顯示「國曆 + 農曆」不同、家人段只顯示農曆民國年 + 閏月註記、對齊全系統 Change C v3 lunar-only 精神）
- **失敗處理**：對話框純前端 state，無 API 失敗風險。

## 情境 8：本次候位已額滿（首頁 + 報名頁兩種入口）

- **使用者目標**：想登記但發現額滿了，需要知道下次什麼時候可以報名。
- **進入點**：首頁直接看到「報名已額滿」卡片，或從首頁進 `/register` 觸發跳轉。
- **操作步驟**：
  1. **首頁路徑**：當 `isFull=true` 時，「我要候位」卡片換成「本次報名已額滿」面板，主按鈕變灰並寫「報名已額滿」（不可按）。
  2. **`/register` 路徑**：進入後 `isFull=true` 觸發 5 秒倒數，自動跳回首頁。
- **預期結果**：使用者看到「本次預約人數已達上限，敬請報名下次開科辦事，下次開科辦事開放報名時間為 YYYY 年 M 月 D 日 中午 12:00 整」（依 `scheduledOpenTime` 或預設算「nextSessionDate + 1 天 12:00」）。
- **邊角案例**：
  - **管理員設定了 `scheduledOpenTime` 自訂時間**：顯示自訂時間（如「2026 年 6 月 5 日 09:00 星期四」）。
  - **沒設 `nextSessionDate` 也沒 `scheduledOpenTime`**：顯示「未設定」，客戶無從得知下次時段。
  - **Change A 後額滿仍佔名額**：即使有人取消，名額不再釋出（除非管理員手動刪除），客戶看到「額滿」就是真的額滿。
- **失敗處理**：5 秒倒數計時器在跳轉前不能讓使用者中斷；若 `navigate('/')` 失敗（極少數），使用者卡在訊息頁，需手動 F5。

## 情境 9：報名後想取消候位（v3 已移除前台入口）

- **使用者目標**：臨時不能去，想取消今天的候位。
- **進入點**：原本是 `/status/:queueNumber` 詳細對話框的「取消預約」按鈕；**Change C 已移除**。
- **操作步驟**：
  1. 使用者進入查詢頁找到自己的候位 → **找不到「取消」按鈕**。
  2. 唯一可行路徑：**致電宮廟 / LINE 告知管理員 → 管理員從後台手動取消**（管理員情境見 admin 文件情境 9）。
- **預期結果**：客戶無法自助取消；防止誤點誤取消。
- **邊角案例**：
  - **客戶看不到自助取消，會抱怨**：宮廟需在 LINE / 公告教育「取消請告知工作人員」。
  - **後端 cancel API 仍存在**：Change A 已加 `_id` + 姓名電話雙重身分驗證，但前台不再呼叫；只供管理員或第三方對接使用。
- **失敗處理**：客戶若強行去找 cancel API URL 也打不通（前台無此呼叫），系統安全側無風險。

## 情境 10：忘了候位號碼，只記得姓名

- **使用者目標**：報完名沒截圖、沒記號碼，現在只記得自己叫什麼。
- **進入點**：`/status/search`。
- **操作步驟**：
  1. 姓名欄填名字，電話欄空著，按查詢。
  2. 系統列出所有符合姓名的候位（同名情況可能多筆）。
- **預期結果**：找到候位記錄 → 可看到號碼與狀態。
- **邊角案例**：
  - **菜市場名 / 同名多筆**：使用者要靠電話 / 詳細資料分辨。
  - **以「家人姓名」查到「主客戶記錄」**：因為後端會搜 `familyMembers.name`，列表會出現該家庭的所有相關記錄。
- **失敗處理**：完全沒登記過 → 紅框「找不到符合條件的候位記錄」。

## 情境 11：忘了候位號碼，只記得電話

- **使用者目標**：同情境 10 但用電話查。
- **進入點**：`/status/search`。
- **操作步驟**：
  1. 電話欄填電話，姓名欄空著，按查詢。
- **預期結果**：列出該電話下所有候位（一個家庭多次報名都顯示）。
- **邊角案例**：
  - 同一電話被不同人借用報名 → 全列出，需靠詳細資料分辨。
- **失敗處理**：電話格式錯 → 後端比對不到 → 「找不到符合條件」。

## 情境 12：填表時 Email 格式錯誤

- **使用者目標**：填寫 email 但格式不對。
- **進入點**：`/register`。
- **操作步驟**：
  1. Email 欄填 `abc@`（缺後段）或 `abc.com`（缺 @）。
  2. 按完成登記。
- **預期結果**：
  - HTML5 `type="email"` 觸發瀏覽器原生驗證提示。
  - 或 backend Joi/yup validator 拒絕並回 400。
- **邊角案例**：
  - Email 是選填，留空可通過。
  - 簡化模式關閉時也仍然「選填」（PRD 規範 email 從來不是必填）。
- **失敗處理**：`formErrors.email` 顯示在欄位下方紅字。

---

## 情境 13：填表時電話格式錯誤

- **使用者目標**：填了非數字、缺位數的電話。
- **進入點**：`/register`。
- **操作步驟**：
  1. 電話填「abc」或「09xxxx」少於 10 碼。
  2. 按完成登記。
- **預期結果**：前端或後端驗證擋下 → `formErrors.phone` 紅字提示。
- **邊角案例**：
  - 市話格式（如 `02-12345678`）系統是否接受？需以後端 schema 為準。
  - 國際碼（如 `+886-9xxx`）非保證支援。
- **失敗處理**：使用者校正後重送。

## 情境 14：填表時忘記填農曆生日

## 情境 24：定時自動開放公開候位登記

- **使用者目標**：客戶事先得知「12:00 開放搶報」，準時上線。

  1. 管理員預先設定 `scheduledOpenTime=YYYY-MM-DD 12:00` 且 `autoOpenEnabled=true`。

  - **時區陷阱**：scheduled time 已存為帶 +08:00 ISO 字串，後端 Node.js timezone 必須對齊台北。
- **失敗處理**：客戶無從得知 scheduler 是否運作，需重整觀察。
